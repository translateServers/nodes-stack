# 大屏设计平台整体演进规划 Spec

## Why

Nebula 已具备较成熟的单用户画布编辑能力，但研究报告提出的数据驱动、非破坏性版本、角色工作区和多人协作仍存在明显缺口。本规划采用小步快跑方法，将整体愿景拆成可独立验收、可发布、可回滚的阶段，避免一次性重构。

## What Changes

- 建立从当前单用户编辑器到数据驱动协作平台的七阶段演进路线。
- 每个阶段只解决一个主要风险或交付一个完整能力闭环。
- 优先修复现有一致性、安全性和测试可信度问题，再增加新能力。
- 将“可靠性与基础访问控制”“单用户交互”“非破坏性配置”“服务端修订版”“权限与工作区”“实时协作”分开实施。
- 统一使用“数据、逻辑、视觉、交互”四层配置、“服务端修订版（revision）”和“本地项目快照”等术语。
- 每一阶段完成后再编写该阶段的细化实施 Spec，整体规划不直接约束具体代码实现细节。
- 不在首轮规划中直接引入 CRDT、复杂分支合并或 AI 自动化；只有前置阶段稳定后才评估。
- 不包含破坏性 API 变更；若后续阶段需要迁移数据，必须提供兼容读取或迁移方案。

## Impact

- Affected specs:
  - `research-interaction-architecture`：作为已完成的基础交互架构参考。
  - 后续按阶段新增的可靠性、工具闭环、数据绑定、版本、角色工作区、协作 Spec。
- Affected code:
  - `apps/web/src/features/screen/`：编辑器、画布、属性面板、工具、数据源、历史和协作界面。
  - `apps/nestjs-server/src/modules/screen/`：项目保存、发布、版本、权限与协作接口。
  - `apps/nestjs-server/prisma/schema/`：版本、成员、数据源或协作相关数据模型。
  - `packages/shared/src/schemas/`：前后端共享契约。
  - `apps/web/e2e/`：编辑器关键路径和多用户场景验证。

## ADDED Requirements

### Requirement: 小步快跑交付原则

系统 SHALL 按原子变更推进，每个实施任务只包含一个可验证结果，并保证任务完成后主干仍可构建、可测试、可发布。

#### Scenario: 原子任务交付

- **WHEN** 开始某一阶段的细分实施
- **THEN** 每个任务只修改一个能力链路或完成一次清晰迁移
- **AND** 新增定义与接入调用在存在风险时拆成两个任务
- **AND** 每个任务提供自动化测试或明确的手动验证方法
- **AND** 不将无关重构和功能开发放入同一任务

#### Scenario: 阶段出口控制

- **WHEN** 一个阶段宣称完成
- **THEN** 该阶段 checklist 全部通过
- **AND** 阶段要求的类型检查、Lint、Biome、单元测试或 E2E 已执行
- **AND** 未完成能力不得仅因存在类型、Hook、占位 UI 或本地状态而标记完成
- **AND** 下一阶段不依赖未通过验收的前置能力

### Requirement: 阶段 0 可靠性与事实基线

系统 SHALL 首先修复影响现有用户和后续演进的安全、一致性、状态同步及测试基线问题，并建立不依赖阶段 4 项目成员模型的最小访问控制基线。

#### Scenario: 公开预览发布状态隔离

- **GIVEN** 一个大屏项目状态为草稿
- **WHEN** 任意访问者通过无需身份认证的公开预览接口请求该项目
- **THEN** 系统不得返回该项目内容
- **AND** 只有已发布项目可通过公开预览接口获取
- **AND** 该规则不授予编辑、发布、删除或受保护详情读取权限

#### Scenario: 受保护项目接口要求身份认证

- **WHEN** 未认证访问者调用项目列表、受保护详情、创建、更新、发布或删除接口
- **THEN** 服务端返回 401
- **AND** 公开预览接口是唯一无需身份认证的项目读取场景
- **AND** 阶段 0 只保证受保护接口需要有效身份，不提前实现阶段 4 的项目成员角色矩阵

#### Scenario: 编辑与预览一致

- **GIVEN** 用户在编辑器中旋转或设置组件公共样式
- **WHEN** 用户打开预览
- **THEN** 预览与编辑器采用同一套公共样式解析规则
- **AND** 位置、尺寸、旋转、透明度、边框和可见性表现一致

#### Scenario: 并发保存基础保护

- **GIVEN** 两个客户端基于同一保存基线打开项目
- **WHEN** 一个客户端已经保存新状态，另一个客户端仍提交旧基线
- **THEN** 服务端拒绝静默覆盖
- **AND** 客户端获得明确冲突提示和重新加载入口

#### Scenario: 属性面板双向同步

- **GIVEN** 用户正在拖动或缩放同一个组件
- **WHEN**组件的位置或尺寸持续变化
- **THEN** 属性面板显示当前值而不是过期局部值
- **AND** 属性输入提交策略明确且有测试覆盖

#### Scenario: 当前测试基线可信

- **WHEN** 阶段 0 完成
- **THEN** 当前实际测试文件、用例数量和命令结果被重新核实
- **AND** 旧文档中的过期测试数字不再作为当前完成证据

### Requirement: 阶段 1 单用户交互闭环

系统 SHALL 在增加平台级能力前，完成现有交互架构的生产接入与主要工具行为闭环。

#### Scenario: 交互状态机成为真实状态源

- **WHEN** 用户在选择、框选、拖拽、缩放、旋转、平移和文本编辑状态之间切换
- **THEN** 画布通过统一交互状态机管理互斥状态
- **AND** 不再由多个可能冲突的布尔状态分别决定同一交互

#### Scenario: 工具行为闭环

- **WHEN** 用户选择文字、矩形、椭圆、图片、吸管或缩放工具
- **THEN** 工具具有对应的可观察画布行为
- **AND** 不允许仅切换 activeTool 和 cursor 就将工具标记完成

#### Scenario: 关键组合交互回归

- **WHEN** 用户执行框选后拖拽、双击进入分组、Alt 拖拽复制、图层排序或快捷键缩放
- **THEN** 浏览器级自动化测试覆盖这些关键路径
- **AND** 浏览器默认行为不会覆盖应用行为

### Requirement: 阶段 2 非破坏性数据与交互分层

系统 SHALL 将组件的“数据、逻辑、视觉、交互”四层配置建立为可独立修改的稳定契约，并提供真实数据源绑定闭环。

#### Scenario: 四层配置独立修改

- **GIVEN** 一个图表组件包含“数据、逻辑、视觉、交互”四层配置
- **WHEN** 用户修改其中任意一层
- **THEN** 未修改层保持不变
- **AND** 共享 Schema 能校验各层配置

#### Scenario: 静态数据源绑定

- **WHEN** 用户为图表配置静态数据源和字段映射
- **THEN** 图表使用数据源解析结果渲染
- **AND** 不再要求用户把数据直接塞入组件展示属性

#### Scenario: API 数据源绑定

- **WHEN** 用户配置受支持的 API 数据源
- **THEN** 系统提供请求测试、响应预览、字段映射、加载、错误和空数据状态
- **AND** 定时刷新可取消且不会产生浮动 Promise
- **AND** 敏感请求头不得以明文方式暴露在公开预览数据中

#### Scenario: 画布配置进入历史

- **WHEN** 用户修改画布尺寸、背景或适配方式
- **THEN** 修改可撤销和重做
- **AND** 历史记录不会只覆盖组件数组

### Requirement: 阶段 3 服务端修订版与恢复

系统 SHALL 在实时协作之前提供服务端权威的不可变修订版历史、差异查看和安全恢复能力。本文中的“修订版”均指服务端 revision，不指本地编辑历史、本地项目快照或阶段 0 的保存基线。

#### Scenario: 保存生成服务端修订版

- **GIVEN** 已认证用户通过受保护项目接口执行保存
- **WHEN** 用户成功保存项目
- **THEN** 服务端生成不可变修订版（revision）
- **AND** 修订版记录操作者身份、时间、基础修订版和变更摘要
- **AND** 阶段 3 复用阶段 0 的身份认证基线，不依赖尚未实施的阶段 4 项目成员角色矩阵

#### Scenario: 比较两个服务端修订版

- **WHEN** 用户选择两个修订版进行比较
- **THEN** 系统展示组件新增、删除、移动和属性修改
- **AND** 能区分画布配置以及数据、逻辑、视觉、交互四层配置变化

#### Scenario: 恢复历史修订版

- **WHEN** 用户恢复历史修订版
- **THEN** 系统创建一个新的当前修订版
- **AND** 不删除原有修订版历史
- **AND** 恢复行为可审计

#### Scenario: 本地项目快照平滑迁移

- **GIVEN** 用户浏览器中已有本地项目快照
- **WHEN** 服务端修订版功能上线
- **THEN** 本地项目快照仍可读取或导入
- **AND** 不因服务端修订版体系直接丢失用户已有本地项目快照

### Requirement: 阶段 4 权限隔离与角色工作区

系统 SHALL 在阶段 0 身份认证基线上建立全局权限与项目成员权限，并为开发者和业务员提供可理解、可执行的角色工作区。项目成员角色统一称为“所有者、编辑者、查看者”；“开发者、业务员”仅表示工作区类型，不直接等同于项目成员权限。

#### Scenario: 身份认证与权限不足分离

- **WHEN** 未认证访问者调用受保护接口
- **THEN** 服务端返回 401
- **WHEN** 已认证用户调用其全局权限或项目成员权限不允许的接口
- **THEN** 服务端返回 403
- **AND** 权限不能仅依赖前端隐藏菜单

#### Scenario: 项目成员权限

- **GIVEN** 项目包含所有者、编辑者和查看者
- **WHEN** 已认证成员读取、修改、发布或删除项目
- **THEN** 服务端按项目成员角色限制操作
- **AND** 非成员通过猜测项目 ID 请求受保护项目资源时不得获知项目内容或存在性
- **AND** 服务端返回不泄露资源存在性的 404
- **AND** 已发布项目仍可通过公开预览接口读取，但公开预览不授予任何项目成员权限

#### Scenario: 角色工作区

- **WHEN** 用户选择开发者工作区或业务员工作区进入同一项目
- **THEN** 系统可提供不同的默认面板组合和功能入口
- **AND** 开发者工作区优先展示数据与逻辑配置
- **AND** 业务员工作区优先展示组件、视觉与基础交互配置
- **AND** 工作区类型只影响默认界面，不扩大用户的全局权限或项目成员权限
- **AND** 工作区布局可保存和恢复

### Requirement: 阶段 5 实时协作基础

系统 SHALL 在服务端修订版与权限隔离基础稳定后，实现可观察、可恢复的实时多人协作基础能力。

#### Scenario: 项目房间与 Presence

- **WHEN** 多个有权限的用户打开同一项目
- **THEN** 用户加入同一协作房间
- **AND** 可看到在线成员、远端光标和远端选区
- **AND** 断线用户的 Presence 会被清理

#### Scenario: 增量同步

- **WHEN** 一个用户移动组件或修改属性
- **THEN** 其他客户端收到增量变更而非完整项目覆盖
- **AND** 服务端维护权威修订版（revision）顺序
- **AND** 断线重连后可根据修订版序号补偿缺失变更

#### Scenario: 评论与批注

- **WHEN** 用户在组件或画布位置添加评论
- **THEN** 其他协作者实时看到评论线程
- **AND** 评论支持回复、解决和重新打开
- **AND** 评论操作遵守项目权限

### Requirement: 阶段 6 分支合并与智能辅助评估

系统 SHALL 将复杂分支合并和 AI 辅助作为后置评估阶段，只有服务端修订版、差异模型和协作链路稳定后才进入实施决策。

#### Scenario: 分支能力进入实施门槛

- **WHEN** 团队评估项目分支和三方合并
- **THEN** 已存在稳定服务端修订版（revision）、结构化 diff 和冲突模型
- **AND** 有真实多角色并行编辑需求证据
- **AND** 不使用整块 JSON 文本冲突作为最终合并体验

#### Scenario: AI 辅助保持可解释

- **WHEN** 系统提供图表、配色或冲突解决建议
- **THEN** AI 输出仅作为可接受或拒绝的建议
- **AND** 不绕过权限、服务端修订版和审计链路自动覆盖项目

### Requirement: 分阶段 Spec 驱动

系统 SHALL 为每个进入实施的阶段创建独立细化 Spec，而不是直接依据本整体规划一次性开发全部内容。

#### Scenario: 启动下一阶段

- **WHEN** 用户决定启动某一阶段
- **THEN** 创建该阶段独立的 `spec.md`、`tasks.md` 和 `checklist.md`
- **AND** 细化原子任务、数据迁移、测试矩阵与回滚边界
- **AND** 只有该阶段规格获批后才开始实现

## MODIFIED Requirements

### Requirement: 现有交互架构完成度声明

`research-interaction-architecture` 中已完成的任务 SHALL 继续作为事件路由、快捷键、智能参考线、图层操作和基础交互的历史实施记录，但“已完成”仅表示对应任务已交付，不表示本整体研究愿景全部实现。

#### Scenario: 区分模块完成与生产接入

- **WHEN** 某个 Hook、类型或纯函数已经存在并通过单元测试
- **THEN** 文档必须区分“模块已实现”与“生产路径已接入”
- **AND** 只有真实用户路径调用该能力后才可标记为产品能力完成

### Requirement: 统一术语

现有 Undo/Redo SHALL 被描述为“本地编辑历史”，浏览器保存的快照 SHALL 被描述为“本地项目快照”，阶段 0 并发保存使用的条件字段 SHALL 被描述为“保存基线”，阶段 3 起由服务端持久化的不可变历史 SHALL 被描述为“服务端修订版（revision）”。以上概念不得混称为版本、分支或多人冲突解决；组件配置层统一称为“数据、逻辑、视觉、交互”四层。

#### Scenario: 文档引用历史与配置概念

- **WHEN** 阶段 Spec、任务或验收项引用编辑历史、项目快照、并发保存条件、服务端历史或组件配置层
- **THEN** 文档使用本 Requirement 定义的统一术语
- **AND** 不使用“版本”同时指代本地编辑历史、本地项目快照、保存基线或服务端修订版（revision）

## REMOVED Requirements

无。本规划不移除现有功能；后续阶段只允许通过兼容迁移逐步替换旧路径。
