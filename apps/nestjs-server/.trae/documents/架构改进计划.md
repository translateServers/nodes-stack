# 架构改进计划

基于单体架构技术分析报告，按优先级分 4 个阶段实施全部已识别问题。

## 阶段一：安全性修复与配置一致性（高优先级）

### 1.1 CORS 配置生效

**问题**：[main.ts](file:///c:/worker/nest-server/src/main.ts) 中 `app.enableCors()` 无参数，而 [app.schema.ts](file:///c:/worker/nest-server/src/config/schemas/app.schema.ts) 定义了 `corsOrigin` 配置项未被消费。

**方案**：
- 修改 `src/main.ts`：从 `TypedConfigService` 读取 `corsOrigin`，传入 `app.enableCors({ origin: corsOrigin })`
- 修改 `src/config/schemas/app.schema.ts`：将 `corsOrigin` 类型从 `z.string()` 改为 `z.string().or(z.array(z.string()))` 支持多 Origin，或保持 string 但支持逗号分隔

### 1.2 API 前缀配置生效

**问题**：[app.schema.ts](file:///c:/worker/nest-server/src/config/schemas/app.schema.ts) 定义 `apiPrefix` 默认 `api/v1`，但 [main.ts](file:///c:/worker/nest-server/src/main.ts) 硬编码 `app.setGlobalPrefix('api')`。

**方案**：
- 修改 `src/main.ts`：从 `TypedConfigService` 读取 `apiPrefix`，传入 `app.setGlobalPrefix(apiPrefix)`
- 同步更新 Swagger 文档路径：`SwaggerModule.setup(\`/${apiPrefix}/docs\`, app, document)`

### 1.3 速率限制

**问题**：登录、验证码等接口无防暴力破解保护。

**方案**：
- 安装 `@nestjs/throttler`
- 创建 `src/common/guards/throttler.guard.ts`：自定义 `ThrottlerGuard`，支持 `@Public()` 跳过（与 JwtAuthGuard 模式一致）
- 创建 `src/common/decorators/skip-throttle.decorator.ts`：`@SkipThrottle()` 装饰器
- 在 `src/app.module.ts` 中注册 `ThrottlerModule.forRoot()` 和 `APP_GUARD` 使用自定义 ThrottlerGuard
- 对登录/验证码接口配置更严格的限流（如 `@Throttle({ default: { limit: 5, ttl: 60000 } })`）

### 1.4 Refresh Token 哈希存储

**问题**：[RefreshToken.prisma](file:///c:/worker/nest-server/prisma/schema/RefreshToken.prisma) 中 `token` 字段明文存储，数据库泄露即导致 Token 被盗用。

**方案**：
- 修改 `src/modules/auth/auth.service.ts`：
  - `generateTokens()` 中存储前对 refreshToken 做 SHA-256 哈希：`crypto.createHash('sha256').update(refreshToken).digest('hex')`
  - `refreshToken()` 中查询时同样哈希后查询
- 修改 `prisma/schema/RefreshToken.prisma`：`token` 字段注释更新为存储哈希值
- 修改 `src/modules/auth/auth.service.ts` 中 `generateDebugToken()`：调试方法中打印原始 token，存储哈希值

### 1.5 验证码存储安全化

**问题**：[captcha.service.ts](file:///c:/worker/nest-server/src/modules/auth/captcha.service.ts) 使用进程内 `Map` 存储，多实例不共享且无上限控制。

**方案**（轻量级，不引入 Redis）：
- 添加 `MAX_CAPTCHA_STORE_SIZE` 常量（如 10000），`generateCaptcha()` 时检查 Map 大小
- 优化 `cleanExpiredCaptchas()`：改为定时清理（`@Injectable()` 实现 `OnModuleInit`，用 `setInterval` 定期清理），而非每次生成时全量遍历
- 在类注释中标注：生产多实例部署需替换为 Redis 存储

### 1.6 DatabaseSchema URL 校验兼容

**问题**：[database.schema.ts](file:///c:/worker/nest-server/src/config/schemas/database.schema.ts) 使用 `z.string().url()`，但 SQLite 的 `file:./dev.db` 不是合法 URL。

**方案**：
- 修改 `src/config/schemas/database.schema.ts`：将 `url: z.string().url()` 改为 `url: z.string().min(1)`，因为不同数据库的连接字符串格式差异大（SQLite 用 `file:`，PostgreSQL 用 `postgresql://`），统一用 URL 校验不合理

## 阶段二：架构扩展与健壮性（高-中优先级）

### 2.1 数据库双模式支持（SQLite 开发 / PostgreSQL 生产）

**问题**：SQLite 不支持并发写入和多实例部署，生产环境需 PostgreSQL。

**方案**：
- 修改 `prisma/schema.prisma`：将 `provider = "sqlite"` 改为根据环境动态选择（Prisma 7 支持通过 `prisma.config.ts` 配置）
- 修改 `prisma.config.ts`：根据 `NODE_ENV` 或 `DATABASE_PROVIDER` 环境变量决定使用哪个 datasource
- 修改 `src/prisma/prisma.service.ts`：
  - 开发环境（SQLite）：继续使用 `PrismaBetterSqlite3` adapter
  - 生产环境（PostgreSQL）：使用标准 PrismaClient 连接（无需 adapter）
  - 通过 `TypedConfigService` 获取数据库类型配置
- 修改 `src/config/schemas/database.schema.ts`：添加 `provider: z.enum(['sqlite', 'postgresql']).default('sqlite')` 字段
- 修改 `src/config/config-loader.ts`：添加 `DATABASE_PROVIDER` 环境变量映射
- 更新 `.env.example`：添加 `DATABASE_PROVIDER` 和两种数据库 URL 示例
- 生成 PostgreSQL migration：`npx prisma migrate dev --name init-postgresql`

### 2.2 RBAC 授权守卫

**问题**：数据模型已定义 Role/Menu/Permission，但无授权守卫实现。

**方案**：
- 创建 `src/common/decorators/roles.decorator.ts`：`@Roles()` 装饰器，通过 `SetMetadata` 设置所需角色
- 创建 `src/common/decorators/permissions.decorator.ts`：`@Permissions()` 装饰器，通过 `SetMetadata` 设置所需权限
- 创建 `src/common/guards/roles.guard.ts`：`RolesGuard`，从 `request.user` 获取用户角色并与 `@Roles()` 元数据比对
- 创建 `src/common/guards/permissions.guard.ts`：`PermissionsGuard`，从用户角色关联的菜单中提取权限列表，与 `@Permissions()` 元数据比对
- 在 `src/app.module.ts` 中注册 `APP_GUARD` 使用 `RolesGuard`（在 JwtAuthGuard 之后）
- 修改 `src/common/interfaces/jwt.interface.ts`：扩展 `JwtPayload` 和 `UserPayload` 包含角色信息
- 修改 `src/modules/auth/strategies/jwt.strategy.ts`：`validate()` 中查询用户角色并附加到 payload

### 2.3 日志系统统一

**问题**：[app-logger.service.ts](file:///c:/worker/nest-server/src/modules/logger/app-logger.service.ts) 继承 `ConsoleLogger`，而 [logger.factory.ts](file:///c:/worker/nest-server/src/modules/logger/logger.factory.ts) 创建 `WinstonModule.createLogger`，两套日志系统并存。

**方案**：
- 删除 `src/modules/logger/app-logger.service.ts`（AppLogger 类）
- 修改 `src/modules/logger/logger.module.ts`：导出 Winston Logger 实例作为统一日志服务
- 修改 `src/main.ts`：使用 `createLogger()` 返回的 Winston Logger 作为 `app.useLogger()`
- 全局搜索替换 `AppLogger` 引用为 NestJS 内置 `Logger`（Winston 已接管底层日志输出）
- 保留 `LogQueryService` 不变

### 2.4 健康检查增强

**问题**：[health.controller.ts](file:///c:/worker/nest-server/src/modules/health/health.controller.ts) 仅返回 `process.uptime()`，未检查数据库连通性。

**方案**：
- 修改 `src/modules/health/health.module.ts`：导入 `PrismaModule`
- 修改 `src/modules/health/health.controller.ts`：注入 `PrismaService`，执行 `$queryRaw\`SELECT 1\`` 检查数据库连通性
- 响应格式：`{ status: 'ok' | 'degraded', timestamp, uptime, database: 'connected' | 'disconnected' }`

### 2.5 Graceful Shutdown

**问题**：未显式注册 `enableShutdownHooks`，进程信号无法触发 `OnModuleDestroy`。

**方案**：
- 修改 `src/main.ts`：在 `bootstrap()` 中添加 `app.enableShutdownHooks()`

## 阶段三：代码质量与可维护性（中优先级）

### 3.1 删除死代码 ZodValidationPipe

**问题**：[zod-validation.pipe.ts](file:///c:/worker/nest-server/src/common/pipes/zod-validation.pipe.ts) 自定义 Pipe 未被使用（全局用的是 nestjs-zod 的），且内部有 `console.log(error)` 调试代码。

**方案**：

- 运行 `pnpm run barrels` 更新索引

### 3.2 TypedConfigService 消除 `any`

**问题**：[typed-config.service.ts](file:///c:/worker/nest-server/src/typed-config.service.ts) 第 28 行 `let result: any = this.config` 违反项目红线规则。

**方案**：
- 修改 `src/config/typed-config.service.ts` 的 `get()` 方法：
  ```typescript
  get<P extends ConfigPath<RootConfig>>(path: P): ConfigPathValue<RootConfig, P> {
    const keys = path.split('.');
    let result: unknown = this.config;
    for (const key of keys) {
      if (result == null) {
        throw new Error(`Configuration path "${path}" is undefined.`);
      }
      result = (result as Record<string, unknown>)[key];
    }
    return result as ConfigPathValue<RootConfig, P>;
  }
  ```
  将 `any` 替换为 `unknown`，利用类型守卫收窄

### 3.3 JwtConfig 类型断言消除

**问题**：[auth.module.ts](file:///c:/worker/nest-server/src/modules/auth/auth.module.ts) 第 13 行 `config.namespace('jwt') as JwtConfig` 不必要的类型断言；[auth.service.ts](file:///c:/worker/nest-server/src/modules/auth/auth.service.ts) 第 126-129 行两次 `as JwtSignOptions['expiresIn']` 断言。

**方案**：
- 修改 `src/config/schemas/jwt.schema.ts`：调整 `JwtConfig` 类型定义，使 `accessTokenTtl` 和 `refreshTokenTtl` 直接为 `NonNullable<JwtSignOptions['expiresIn']>` 类型，而非先定义再 Omit
- 修改 `src/modules/auth/auth.module.ts`：移除 `as JwtConfig` 断言，`namespace('jwt')` 返回值应已具备正确类型
- 修改 `src/modules/auth/auth.service.ts`：移除 `as JwtSignOptions['expiresIn']` 断言

### 3.4 SharedModule 处理

**问题**：[shared.module.ts](file:///c:/worker/nest-server/src/modules/shared/shared.module.ts) 为空壳，无任何 provider。

**方案**：
- 删除 `src/modules/shared/shared.module.ts`
- 从 `src/app.module.ts` 的 imports 中移除 `SharedModule`
- 删除相关索引导出

### 3.5 Debug Token 方法清理

**问题**：[auth.service.ts](file:///c:/worker/nest-server/src/modules/auth/auth.service.ts) 中 `generateDebugToken()` 方法存在于生产代码中。

**方案**：

- 删除 `AuthService` 中的 `generateDebugToken` 方法

## 阶段四：基础设施与部署（中-低优先级）

### 4.1 缓存层集成

**问题**：无 Redis 或内存缓存，高频读取场景需重复查询数据库。

**方案**：
- 安装 `@nestjs/cache-manager` 和 `cache-manager`
- 创建 `src/modules/cache/cache.module.ts`：`CacheModule` 封装缓存配置
- 开发环境使用内存缓存，生产环境可替换为 Redis store
- 在字典查询等高频只读接口上添加 `@CacheKey()` + `@CacheTTL()` 装饰器
- 在字典写操作后手动清除缓存

### 4.2 Docker 部署配置

**问题**：无容器化配置。

**方案**：
- 创建 `Dockerfile`：多阶段构建（build 阶段 + runtime 阶段）
- 创建 `docker-compose.yml`：定义 app + postgresql 服务
- 创建 `.dockerignore`

### 4.3 Swagger 生产环境控制

**问题**：Swagger UI 在生产环境暴露。

**方案**：
- 修改 `src/main.ts`：根据 `nodeEnv` 条件性设置 Swagger，生产环境跳过或通过独立配置项控制
- 在 `src/config/schemas/app.schema.ts` 添加 `enableSwagger: z.boolean().default(true)` 配置项
